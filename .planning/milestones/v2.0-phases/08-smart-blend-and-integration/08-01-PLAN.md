  <action>
    Create `packages/server/src/ranking/` directory with four files:

    **normalize.ts** -- Per-source percentile engagement normalization:
    - Define a `PRIMARY_METRIC` map: reddit -> 'upvotes', youtube -> 'views', tumblr -> 'notes', bluesky -> 'likes'. RSS/unknown sources return null.
    - `normalizeEngagement(items)` groups items by `source`, extracts primary metric from `engagementStats` JSON, sorts ascending within group, assigns percentile rank = `position / (count - 1)` (or 0.5 if only 1 item). Items with null engagement data get 0.5 (neutral, per user decision).
    - Returns `Map<number, number>` (item.id -> 0-1 percentile score).
    - Items need: `id: number`, `source: string`, `engagementStats: Record<string, number> | null`.

    **scoring.ts** -- Multi-signal blend score:
    - Weights: recency 0.40, engagement 0.35, contentTypeVariety 0.10, sourceDiversity 0.15 (balanced blend per user decision).
    - `recencyScore(publishedMs, now)`: exponential decay `Math.exp(-ageHours / 8)`. This gives ~0.47 at 6 hours (noticeable drop per user decision), ~0.05 at 24 hours.
    - `contentTypeVarietyScore(contentType, typeCounts, totalItems)`: inverse frequency `1 - (count / totalItems)`. Rare types score higher.
    - `sourceDiversityScore(source, sourceCounts, totalItems)`: inverse frequency `1 - (count / totalItems)`. Underrepresented sources score higher.
    - `computeBlendScore(item, engagementPercentile, typeCounts, sourceCounts, totalItems, now, boostFactor)`: weighted sum of all signals, multiplied by boostFactor (default 1.0).
    - IMPORTANT: The `publishedAt` field from Drizzle may be a Date object or raw epoch seconds. Always normalize to milliseconds before computing age. Check: `item.publishedAt instanceof Date ? item.publishedAt.getTime() : (item.publishedAt as number) * 1000`.

    **interleave.ts** -- Post-sort diversity interleaving:
    - `interleaveBySource(items, maxConsecutive = 2)`: reorders scored+sorted items so no more than `maxConsecutive` consecutive items share the same source.
    - Algorithm: iterate remaining items (already sorted by blendScore desc). For each position, check if last `maxConsecutive` items in result share same source. If so, find highest-scored item from a different source. If no different-source item exists, just append (graceful degradation).
    - Generic over T extending `{ source: string; blendScore: number }`.

    **index.ts** -- rankFeed() orchestrator:
    - `rankFeed<T extends RankableItem>(items, boostMap)`:
      1. Call `normalizeEngagement(items)` to get percentile scores.
      2. Build `typeCounts` and `sourceCounts` maps from item list.
      3. Score each item via `computeBlendScore()`, passing boost from `boostMap.get(item.sourceDetail) ?? 1.0`.
      4. Sort by blendScore descending.
      5. Apply `interleaveBySource()` with maxConsecutive=2.
      6. Return items in new order.
    - Export `RankableItem` interface: `{ id: number, source: string, sourceDetail: string, contentType: string | null, engagementStats: Record<string, number> | null, publishedAt: Date | number }` (plus `[key: string]: unknown` for passthrough).

    **sources.ts** -- Add boost field:
    - Add `boost?: number` to the `ScrapingSource` interface (optional, default 1.0).
    - Add `boost: 1.5` to the `tumblr-bts-trans` source entry in `getBtsScrapingConfig()`. This is the only confirmed fan translation account. Other sources keep default 1.0 (no boost field needed).

  <action>
    Update the GET /feed handler in `packages/server/src/routes/feed.ts` to use `rankFeed()` for ordering instead of `orderBy(desc(contentItems.scrapedAt))`.

    **Pagination strategy change (per research Pitfall 3):**
    The current cursor-based pagination on item ID breaks after re-ranking because items are no longer in ID-descending order. Switch to offset-based pagination for the ranked feed:
    - Add `page` query param (default "1"). Parse as integer, clamp to >= 1.
    - Keep `limit` query param (default 50, clamped 1-100).
    - Keep backward-compatible `cursor` support: if `cursor` is provided but `page` is not, fall back to the old cursor-based logic (so existing clients don't break). But if `page` is provided, use offset-based pagination.
    - For offset-based: `offset = (page - 1) * limit`.

    **Feed query approach:**
    1. Fetch a larger candidate set from the DB: up to 500 approved, non-deleted items (with source/contentType filters if provided). Order by `desc(contentItems.scrapedAt)` for the DB query to get recent items. Remove the old `limit + 1` trick.
    2. Build boost map from `getBtsScrapingConfig()` at route registration time (not per-request): iterate `config.sources`, build `Map<string, number>` of `label -> boost` for sources where `boost` is defined and > 1.0.
    3. Call `rankFeed(candidateItems, boostMap)` to get ranked order.
    4. Slice the ranked result for the requested page: `ranked.slice(offset, offset + limit)`.
    5. Compute `hasMore = offset + limit < ranked.length`.
    6. `nextCursor` for backward compat: if using page-based, set to `null` (clients use page param instead). If using cursor fallback, keep existing behavior.
    7. `total` remains the count of all matching approved items.

    **Important:** The DB row -> FeedItem mapping stays the same. The `rankFeed()` call operates on the DB rows (which have `id`, `source`, `sourceDetail`, `contentType`, `engagementStats` as JSON string, `publishedAt`). Parse `engagementStats` from JSON string to object BEFORE passing to rankFeed. After ranking, map to API FeedItem shape as before.
