1. Create `packages/server/src/pipeline/budget.ts`:
   - `getDailySpend(db: Db, date: string): number` — queries pipeline_runs where startedAt is on `date` (YYYY-MM-DD), sums estimatedCost (parse from text to float). Return 0 if no runs.
   - `getMonthlySpend(db: Db): number` — queries pipeline_runs where startedAt is in current month, sums estimatedCost.
   - `shouldUseFallback(db: Db): boolean` — reads `LLM_DAILY_BUDGET` env var (default '1.00') and `LLM_MONTHLY_BUDGET` env var (default '10.00'). Returns true if daily or monthly spend >= limit.
   - `recordCost(db: Db, runId: number, inputTokens: number, outputTokens: number, costPerInput: number, costPerOutput: number)` — updates the pipeline_runs row with token counts and estimated cost.

2. Create `packages/server/src/pipeline/fallback.ts`:
   - Import `getBtsScrapingConfig` from `@bts/shared/config/sources.js` (or accept config as parameter)
   - `handleFallback(db: Db, items: {id: number, sourceDetail: string}[])` function:
     - For each item, look up the source config to check `needsFilter`
     - If `needsFilter === false` (BTS-focused source like r/bangtan, BANGTANTV, bts-trans): set moderation_status = 'approved', content_type = null (no classification in fallback)
     - If `needsFilter === true` (broad source like r/kpop, Soompi): set moderation_status = 'pending' (queued until LLM is available or budget resets)
     - Log how many items were auto-approved vs queued
   - Build a lookup Set of `sourceDetail` values where `needsFilter === false` from the scraping config for O(1) checks.

3. Create `packages/server/src/pipeline/pipeline.ts`:
   - Import: `generateText`, `Output` from 'ai'; `getProvider` from './provider.js'; `BatchResultSchema` from './schemas.js'; `buildBatchPrompt` from './prompts.js'; `shouldUseFallback`, `recordCost` from './budget.js'; `handleFallback` from './fallback.js'; DB schema tables; drizzle operators.
   - `const BATCH_SIZE = 15;` (midpoint of 10-20 range per user decision)
   - `runPipeline(db: Db)` async function:
     a. Check `shouldUseFallback(db)`. If true:
        - Select all items with moderation_status = 'raw' and deletedAt IS NULL
        - Call `handleFallback(db, items)`
        - Record a pipeline_run with status = 'fallback', fallbackMode = true
        - Return early
     b. Select items with `moderation_status = 'raw'` and `deletedAt IS NULL`, limit `BATCH_SIZE * 5` (up to 75 items per pipeline run)
     c. If no items, return early (nothing to process)
     d. **Atomically** mark selected items as 'pending' — UPDATE moderation_status = 'pending' WHERE id IN (...). This prevents double-processing if another pipeline run starts (same guard pattern as scrape endpoint).
     e. Create a pipeline_runs row with status = 'running', startedAt = now, provider = `${providerConfig.name}`
     f. Process items in batches of BATCH_SIZE:
        - Build prompt via `buildBatchPrompt(batchItems.map((item, i) => ({ index: i, title: item.title, source: item.sourceDetail })))`
        - Call `generateText({ model: provider.model, output: Output.object({ schema: BatchResultSchema }), prompt })` wrapped in try/catch
        - On success: iterate `output.items`, for each decision:
          - If `!relevant || !safe` → set moderation_status = 'rejected', moderatedAt = now
          - If `relevant && safe` → set moderation_status = 'approved', contentType = decision.contentType, moderatedAt = now
          - Insert a pipeline_decisions row for each item
          - Track tokens from `usage.promptTokens` and `usage.completionTokens`
        - On `NoObjectGeneratedError` or any LLM error: auto-approve the entire batch as fallback (log warning). Set moderation_status = 'approved' for all items in the failed batch. This prevents items from being stuck as 'pending'.
     g. After all batches: update pipeline_runs with final stats (itemsProcessed, itemsApproved, itemsRejected, tokens, estimatedCost, status = 'success', completedAt = now)
     h. Clean up pipeline_decisions older than 30 days (hard delete, per user decision)
   - Export `runPipeline`

   **Important anti-patterns to avoid (from RESEARCH.md):**
   - Do NOT make separate LLM calls for relevance, moderation, classification — combine all three in single structured output
   - Do NOT call LLM per-item — always batch
   - Do NOT block the scrape cycle waiting for the pipeline
