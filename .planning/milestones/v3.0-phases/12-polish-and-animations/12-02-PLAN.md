  <action>
    **1. CSS keyframes and class in App.css:**

    Add after the snap skeleton styles (at end of snap section):

    ```css
    /* === Snap Card Entrance Animation === */
    @keyframes snap-card-enter {
      from {
        opacity: 0;
        transform: translateY(30px);
      }
      to {
        opacity: 1;
        transform: translateY(0);
      }
    }

    .snap-card-enter {
      animation: snap-card-enter 250ms cubic-bezier(0.0, 0.0, 0.2, 1) both;
    }

    /* Respect prefers-reduced-motion: disable slide animation, keep instant fade */
    @media (prefers-reduced-motion: reduce) {
      .snap-card-enter {
        animation: none;
        opacity: 1;
        transform: none;
      }
    }
    ```

    **Key CSS details:**
    - `cubic-bezier(0.0, 0.0, 0.2, 1)` is Material Design's "decelerate" easing -- starts fast, slows to a stop. This gives the snappy, energetic feel per user decision.
    - `animation-fill-mode: both` (via `both` shorthand) ensures the element starts at the `from` state and stays at the `to` state after animation completes.
    - Only `transform` and `opacity` are animated -- both are GPU-composited, zero layout/paint cost.
    - `prefers-reduced-motion: reduce` disables the animation entirely, element renders at final state immediately.

    **2. Apply entrance class in SnapFeed.tsx:**

    In the `visibleItems.map()` render block, add the `snap-card-enter` class ONLY to cards where `vi.position === 0` (the active/current card):

    Change the className of the `.snap-card` div from:
    ```tsx
    className="snap-card"
    ```
    to:
    ```tsx
    className={`snap-card${vi.position === 0 ? ' snap-card-enter' : ''}`}
    ```

    **Why this works without additional state:**
    - The key `${vi.realIndex}-${vi.position}` changes when a new card snaps to position 0
    - React unmounts the old card at that key and mounts a new one
    - Mounting a new DOM element re-triggers the CSS animation naturally
    - Parent re-renders that don't change `currentIndex` keep the same keys, so animation does NOT re-trigger on filter/sort changes
    - Scrolling back to a previous card also changes the realIndex at position 0, triggering animation again (per user decision: "including when scrolling back to previous cards")
    - First card on page load: it mounts fresh, animation plays (per user decision: "First card on page load also animates in")

    **Do NOT:**
    - Add animation to cards at position !== 0 (they're off-screen, animating them causes jarring triple-animation)
    - Use useState or useEffect to trigger the animation (the key-based re-mount handles it naturally)

<task type="auto">
  <name>Task 2: Add control bar initial slide-down animation</name>
  <files>
    packages/frontend/src/components/snap/SnapControlBar.tsx

  </files>
  <action>
    The control bar already has a `transition: transform 0.3s` for show/hide behavior. On initial page load, we want it to slide down from off-screen into its visible position, coordinated with the card entrance animation.

    **1. SnapControlBar.tsx changes:**

    Add a `useEffect` + `useState` pattern to start with an initial-hidden state and remove it after mount:

    ```tsx
    const [initialLoad, setInitialLoad] = useState(true);

    useEffect(() => {
      // Small delay so the slide-down is visible (not instant on first paint)
      const timer = requestAnimationFrame(() => {
        setInitialLoad(false);
      });
      return () => cancelAnimationFrame(timer);
    }, []);
    ```

    Update the className to include the initial state:
    - Currently: `className={`snap-control-bar${visible ? '' : ' hidden'}`}`
    - Change to: `className={`snap-control-bar${initialLoad ? ' snap-control-bar-initial' : ''}${visible ? '' : ' hidden'}`}`

    The `initialLoad` state starts true, causing the bar to render off-screen. After one animation frame, it becomes false, and the existing CSS transition slides the bar down.

    **2. CSS in App.css:**

    Add after the `.snap-control-bar.hidden` rule:

    ```css
    .snap-control-bar.snap-control-bar-initial {
      transform: translateY(-100%);
    }
    ```

    This uses the SAME `transition: transform 0.3s cubic-bezier(0.32, 0.72, 0, 1)` already on `.snap-control-bar`. When `.snap-control-bar-initial` is removed (initialLoad becomes false), the existing transition animates the bar from translateY(-100%) to translateY(0). No new transition property needed -- no double-animation risk.

    **Why requestAnimationFrame instead of setTimeout:**
    - RAF fires after the browser has painted the initial frame (bar off-screen), then triggers the state change which causes the transition
    - Avoids arbitrary timeout values
    - Coordinates with the browser's rendering pipeline

    **Reduced motion consideration:** The control bar slide-down is a 0.3s translateY transition, which is subtle. Per W3C guidance, this is borderline -- it's a small, directional movement. However, to be safe, add to the `prefers-reduced-motion: reduce` media query block (extending the one from Task 1):

    ```css
    @media (prefers-reduced-motion: reduce) {
      .snap-card-enter {
        animation: none;
        opacity: 1;
        transform: none;
      }
      .snap-control-bar {
        transition-duration: 0.01ms;
      }
    }
    ```

    This makes the control bar transition effectively instant for reduced-motion users while keeping the visual state correct.
