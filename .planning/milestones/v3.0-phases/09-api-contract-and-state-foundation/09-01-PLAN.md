<objective>
Add server-side sort support to the /api/feed endpoint and propagate the sort parameter through the frontend API client and feed service.

Purpose: PERF-02 requires sort to be computed server-side. The server has the full 500-item candidate set and the ranking pipeline. Client-side sort on a paginated subset gives inconsistent results.

Output: The /api/feed endpoint accepts a `sort` query parameter with values `recommended|newest|oldest|popular|discussed`. The frontend `fetchApiFeed` and `fetchFeed` functions pass the sort parameter through to the server.
</objective>

  <action>
    1. In `packages/shared/src/types/feed.ts`:
       - Add: `export type SortMode = 'recommended' | 'newest' | 'oldest' | 'popular' | 'discussed';`
       - Update `FeedQuery` interface to add `sort?: SortMode;`

    2. In `packages/server/src/routes/feed.ts`:
       - Add `sort?: string` to the Querystring type alongside the existing params.
       - In the page-based ranked feed section (after `const candidateRows = ...`), extract and validate the sort param:
         ```
         const validSorts = ['recommended', 'newest', 'oldest', 'popular', 'discussed'] as const;
         const sortMode = validSorts.includes(request.query.sort as any)
           ? (request.query.sort as typeof validSorts[number])
           : 'recommended';
         ```
       - If `sortMode === 'recommended'`, use the existing `rankFeed()` pipeline (no change).
       - Otherwise, sort `candidateRows` in-memory BEFORE pagination:
         - `newest`: sort by publishedAt descending. Handle the Drizzle timestamp format: `row.publishedAt` may be a Date or epoch seconds integer. Normalize using the same pattern the existing code uses: `row.publishedAt instanceof Date ? row.publishedAt.getTime() : (row.publishedAt as unknown as number) * 1000`.
         - `oldest`: same as newest but ascending.
         - `popular`: parse `row.engagementStats` JSON string, sum all numeric values (upvotes, views, likes, notes), sort descending. For null/missing stats, treat as 0.
         - `discussed`: sort by `row.commentCount` descending.
       - For non-recommended sorts, skip the `rankFeed()` call entirely. Parse engagementStats JSON only for 'popular' sort (others don't need it).
       - After sorting, paginate the sorted array using the same `offset`/`limit` logic already in place.
       - Map results to FeedItem using the inline mapping already used in the ranked path (not `mapRowToFeedItem`, since engagementStats for 'popular' is already parsed).

    Do NOT modify the cursor-based fallback path. The sort param only applies to the page-based path.
  </action>
  <verify>
    - `npx tsc --noEmit -p packages/shared/tsconfig.json` passes (shared types compile)
    - `npx tsc --noEmit -p packages/server/tsconfig.json` passes (server compiles)
    - Start the server and test with curl:
      - `curl "http://localhost:3001/api/feed?page=1&limit=5"` returns items (recommended, existing behavior)
      - `curl "http://localhost:3001/api/feed?page=1&limit=5&sort=newest"` returns items sorted by date descending
      - `curl "http://localhost:3001/api/feed?page=1&limit=5&sort=oldest"` returns items sorted by date ascending
      - `curl "http://localhost:3001/api/feed?page=1&limit=5&sort=popular"` returns items
      - `curl "http://localhost:3001/api/feed?page=1&limit=5&sort=discussed"` returns items sorted by commentCount descending
      - `curl "http://localhost:3001/api/feed?page=1&limit=5&sort=garbage"` returns items (falls back to recommended)
  </verify>
