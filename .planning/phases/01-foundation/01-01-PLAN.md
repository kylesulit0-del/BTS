<task type="auto">
  <name>Task 2: Refactor CORS proxy to parallel failover with Promise.any()</name>
  <files>
    src/utils/corsProxy.ts
  </files>
  <action>
Rewrite `src/utils/corsProxy.ts` to use `Promise.any()` instead of the current sequential `for` loop:

1. Keep the existing PROXIES array (same 3 proxies, same URL builders).

2. Replace the `fetchWithProxy` function body:
   - Create a single `AbortController` shared across all proxy attempts.
   - Use `Promise.any()` to fire all proxy fetch requests simultaneously.
   - Each promise: build proxy URL, call `fetch()` with the shared AbortController signal, check `res.ok`, call `res.text()` to fully read the body.
   - After `Promise.any()` resolves (first success wins, body fully read), call `controller.abort()` to cancel remaining in-flight requests.
   - On `AggregateError` (all proxies failed), throw `new Error("All proxies failed for {url}")`.

3. Keep the per-request timeout behavior: use `AbortSignal.any([controller.signal, AbortSignal.timeout(7000)])` for each fetch so individual requests still timeout after 7 seconds. This requires combining the shared abort signal with the per-request timeout. `AbortSignal.any()` is supported in ES2022 target. If `AbortSignal.any` is not available in the target environment, fall back to just the shared controller signal with a separate setTimeout that calls controller.abort().

4. Preserve the same function signature: `fetchWithProxy(url: string): Promise<string>`.

Important: The `.text()` call must complete BEFORE `controller.abort()` fires. The pattern in RESEARCH.md handles this correctly -- the await inside the map callback fully resolves the text before returning.
  </action>

    - `npm run build` succeeds (Vite build completes without errors)
  </verify>
