  <name>Task 1: Create source registry and per-source fetcher modules</name>
  <files>

  <action>
1. **`src/services/sources/registry.ts`** -- The source registry:
   - Export `SourceFetcher` type: `(source: SourceEntry) => Promise<FeedItem[]>`
   - Export `registerFetcher(type: string, fetcher: SourceFetcher): void` -- adds fetcher to internal Map
   - Export `getFetcher(type: string): SourceFetcher | undefined` -- retrieves fetcher by type
   - At the bottom, import all fetcher modules to trigger self-registration:
     ```
     import "./reddit";
     import "./youtube";
     import "./rss";
     import "./twitter";
     ```
   - Import types from `../../config/types` (SourceEntry) and `../../types/feed` (FeedItem).

2. **`src/services/sources/reddit.ts`** -- Reddit fetcher:
   - Port `fetchSubreddit` logic from current `feeds.ts`.
   - Import `getConfig` from `../../config` to read `config.keywords` for filtering (replaces hardcoded `BTS_KEYWORDS`).
   - Import `stripToText` from `../../utils/sanitize` (replaces the XSS-vulnerable `stripHtml`).
   - Import `fetchWithProxy` from `../../utils/corsProxy`.
   - The fetcher function takes a `SourceEntry` and returns `Promise<FeedItem[]>`.
   - Build Reddit URL from `source.url` (subreddit name) and `source.fetchCount` (default 15).
   - Filter by `config.keywords` when `source.needsFilter` is true.
   - Call `registerFetcher("reddit", fetchRedditSource)` at module scope.
   - Preserve all existing behavior: skip stickied posts, build Reddit URLs, extract thumbnails, format authors.

3. **`src/services/sources/youtube.ts`** -- YouTube fetcher:
   - Port `fetchYouTubeChannel` logic from current `feeds.ts`.
   - Read `config.keywords` from config for filtering.
   - Source entry `url` field contains the channel ID. `label` is the channel name.
   - Use `parseAtom` from `../../utils/xmlParser` (unchanged).
   - Use `fetchWithProxy` from `../../utils/corsProxy`.
   - Call `registerFetcher("youtube", fetchYouTubeSource)`.
   - Preserve existing behavior: video ID extraction, thumbnail generation, Atom feed parsing.

4. **`src/services/sources/rss.ts`** -- Generic RSS fetcher (handles Soompi, AllKPop, any future RSS):
   - Port `fetchNews` and `fetchAllKPop` logic into a single generic RSS fetcher.
   - Takes a SourceEntry where `url` is the RSS feed URL and `label` is the source name.
   - Use `parseRSS` from `../../utils/xmlParser`.
   - Use `stripToText` from `../../utils/sanitize` for content preview (replaces `stripHtml`).
   - Use `sanitizeHtml` from `../../utils/sanitize` for rich content (store clean HTML).
   - Filter by `config.keywords` when `source.needsFilter` is true.
   - Generate IDs using source.id to avoid collisions: `${source.id}-${index}-${Date.now()}`.
   - Call `registerFetcher("rss", fetchRssSource)`.

5. **`src/services/sources/twitter.ts`** -- Twitter/Nitter fetcher:
   - Port `fetchTwitter` logic from current `feeds.ts`.
   - Source entry `url` contains the Nitter search URL.
   - Use `stripToText` from `../../utils/sanitize` (replaces `stripHtml`).
   - Use `fetchWithProxy` from `../../utils/corsProxy`.
   - Call `registerFetcher("twitter", fetchTwitterSource)`.
   - Preserve existing HTML scraping logic (regex-based parsing of Nitter HTML).

Per user decision on source registry: "Registry handles error handling/retries uniformly for all sources -- fetchers just fetch." The individual fetcher functions should NOT wrap themselves in try/catch -- they throw on error, and the orchestrator in feeds.ts handles errors uniformly.

However, note the current RSS fetchers (fetchNews, fetchAllKPop) DO have try/catch returning []. Move this error handling to the orchestrator instead. Each fetcher should just throw if something goes wrong.

Also per user decision: "All fetchers return a shared FeedItem type normalizing data across sources."

<task type="auto">
  <name>Task 2: Refactor feeds.ts orchestrator to use registry and config</name>
  <files>
    src/services/feeds.ts

  <action>
1. **Refactor `src/services/feeds.ts`**:
   - Remove ALL existing fetch functions (fetchReddit, fetchYouTube, fetchNews, fetchAllKPop, fetchTwitter, fetchSubreddit, fetchYouTubeChannel).
   - Remove `BTS_KEYWORDS` regex and `stripHtml` function.
   - Import `getConfig` from `../config`.
   - Import `getFetcher` from `./sources/registry` (this import triggers all fetcher registrations via registry.ts side-effect imports).
   - Rewrite `fetchAllFeedsIncremental(onItems: FeedCallback)`:
     - Read `config.sources` to get all configured source entries.
     - For each source entry, look up `getFetcher(source.type)`.
     - Skip sources with no registered fetcher (log a warning to console).
     - Fire all source fetches in parallel, delivering results incrementally via `onItems` as each resolves -- PRESERVE the exact same progressive loading pattern as the current implementation.
     - Each source fetch is wrapped in error handling: catch errors and return [] (silent failure per user decision on proxy failure UX).
     - The `allItems` array accumulates results, sorted by timestamp descending, and is passed to `onItems` after each source resolves.
   - Rewrite `fetchAllFeeds()` similarly but without incremental callback -- just returns the final sorted array.
   - Keep the `FeedCallback` type export.

2. **Update `src/types/feed.ts`**:
   - Update `FeedSource` type to include "rss" alongside existing types: `"reddit" | "youtube" | "news" | "rss" | "twitter"`. Keep "news" for backward compatibility with existing cached data -- the RSS fetcher produces items with source "rss" but old cached items may have "news".
   - Remove `MEMBER_KEYWORDS` constant (now lives in config as member aliases + compiled keywords regex).
   - Remove `BiasId` type (will be derived from config in Plan 04). Actually, KEEP BiasId for now -- it's used by useBias hook and BiasFilter component which haven't been migrated yet. Mark with a TODO comment that this moves to config-derived type in Plan 04.
   - Keep `FeedItem` interface unchanged.

CRITICAL: Preserve the progressive loading behavior. The current pattern fires all sources and calls `onItems` with the accumulated sorted array as each source resolves. The new implementation MUST do the same -- do not wait for all sources before calling onItems.

CRITICAL: The existing `fetchAllFeedsIncremental` slices results per source (e.g., reddit returns max 20, youtube max 15). This per-source limiting is now handled by `source.fetchCount` in the fetcher modules. Verify the new implementation respects these limits.
  </action>
