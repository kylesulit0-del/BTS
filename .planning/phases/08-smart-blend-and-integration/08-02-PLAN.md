  <files>

  <action>
    **api.ts** -- Server API client with type mapping:

    Create `packages/frontend/src/services/api.ts`:
    - Read `VITE_API_URL` from `import.meta.env.VITE_API_URL`.
    - `isApiMode()`: returns `true` if `VITE_API_URL` is a non-empty string.
    - `mapApiFeedItem(item)`: maps server `FeedItem` (from `@bts/shared/types/feed.js`) to frontend `FeedItem` (from `../types/feed`). Key field mappings:
      - `id`: `String(item.id)` (number -> string)
      - `title`: pass through
      - `url`: pass through
      - `source`: pass through
      - `sourceName`: `item.sourceDetail` (sourceDetail -> sourceName)
      - `timestamp`: `new Date(item.publishedAt).getTime()` (ISO string -> epoch ms)
      - `thumbnail`: `item.thumbnailUrl ?? undefined` (null -> undefined)
      - `stats`: map from `item.engagementStats` -- extract `upvotes`, `comments`, `views`, `likes`, `notes` if present. If `engagementStats` is null, set `stats` to undefined.
      - `contentType`: `item.contentType ?? undefined`
      - `preview`: undefined (server items don't have preview text)
    - `fetchApiFeed(params?)`: fetches from `${VITE_API_URL}/api/feed` with optional query params: `page`, `limit`, `source`, `contentType`. Returns `{ items: FeedItem[], hasMore: boolean, total: number }`. Maps each server item through `mapApiFeedItem()`. Throws on non-OK response.
    - Import the server FeedItem type as a renamed import to avoid collision: `import type { FeedItem as ApiFeedItem, FeedResponse } from '@bts/shared/types/feed.js'` and `import type { FeedItem } from '../types/feed'`.

    **feedService.ts** -- Dual-mode orchestrator:

    Create `packages/frontend/src/services/feedService.ts`:
    - `fetchFeed(options?)`: async function that:
      1. If `isApiMode()` is true: try `fetchApiFeed()` to get items. If it succeeds, return the items. If it fails (network error, non-OK response), log a warning and fall through to client-side.
      2. Fall through or if `isApiMode()` is false: call `fetchAllFeedsIncremental(onItems)` from `./feeds` for client-side fetching.
    - `options` parameter: `{ force?: boolean, onItems?: (items: FeedItem[]) => void, source?: string, contentType?: string }`.
    - In API mode, `onItems` callback is called once with the full result (no incremental loading since server returns complete ranked set). In client-side mode, `onItems` is passed through to `fetchAllFeedsIncremental` for progressive loading.
    - Export `isApiMode` from api.ts for consumers who need to check.
    - The auto-fallback should be silent -- no error UI if API fails but client-side works. Only show error if both modes fail.
  </action>
  <verify>

    Verify both files exist and export their main functions.

    - Change the `load()` function to call `fetchFeed({ force, onItems: (items) => setAllItems(items), source: sourceFilter, contentType: typeFilter })` instead of `fetchAllFeedsIncremental(...)` directly.
    - In API mode, source and contentType filtering happens server-side (passed as query params). In client-side mode, filtering happens locally as before.
    - The hook should detect API mode via `isApiMode()` (import from `../services/api`):
      - API mode: pass `source` and `contentType` to `fetchFeed()`. Server returns pre-filtered, pre-ranked items. Local bias filtering still applies (server doesn't know about bias preferences).
      - Client-side mode: pass no source/contentType to `fetchFeed()` (filtering done locally after fetch as before).
    - The hook parameters stay the same: `useFeed(filter, biases)`. Caller API unchanged.
    - Keep existing cache logic -- in API mode, cache the fetched items in localStorage same as client-side mode.
    - Keep existing retry logic -- if API mode fetch fails and falls back to client-side, the retry mechanism handles the client-side case.
    - IMPORTANT: Update the `load()` useCallback dependency array appropriately if referencing filter/contentType inside.

    NOTE: The hook currently accepts `filter: FeedSource | "all"` for source filtering and applies it post-fetch. In API mode, this should be sent as a query param instead. Keep the local filtering for client-side mode. Use a conditional: if `isApiMode()`, pass filter to fetchFeed; otherwise apply locally as before. For biases, always filter locally (both modes).

    **feeds.ts** -- Add basic diversity interleaving to client-side pipeline:

    Per user decision, client-side fallback should use "basic blend (diversity + recency)" not just recency+engagement. Add a simple interleaving function and integrate it into `applyFeedPipeline()`:

    Add `interleaveSimple(items: FeedItem[]): FeedItem[]` function:
    - After the existing sort by `computeFeedScore`, reorder so no more than 2 consecutive items share the same source.
    - Algorithm: iterate through remaining items. If last 2 result items have same source, find next item from a different source. If none, just append.
    - This is a simpler version of the server's interleave -- no full normalization, just diversity post-sort.

    Update `applyFeedPipeline()` to call `interleaveSimple()` after the final sort:
    ```
    return interleaveSimple(capped.sort(...));
    ```
